# Phase 3 — 自动修复框架设计

> 版本: 1.0 | 日期: 2026-06-07

## 核心原则

渐进过渡：先写 recipe（带触发条件和决策点标记），预留 auto-safe 扩展空间。

## 修复等级

| 等级 | 定义 | 示例 | 策略 |
|:----:|:----|:-----|:----|
| **L1** 无感修复 | 用户无感知，无服务影响 | 改配置、清缓存 | 完全自动 |
| **L2** 有感无风险 | 短暂/非关键服务影响 | 重启非关键服务 | 自动 + 事后通知 |
| **L3** 不可逆 | 数据丢失/服务中断风险 | 删数据、改 token、重启关键服务 | 必须你批准 |

## 修复流程

```
出问题 → diagnose.sh 并行诊断
  ├─ 有匹配 → 判断修复等级 → L1/L2 自动 / L3 找你
  └─ 无匹配 → 找你排查 → 记录 → 两次出现则固化
每个修复步骤后 → 验证探针 → 正常则 ✅ / 更糟→回滚/通知
```

## 锁机制

锁文件: `/tmp/astra-sre-lock-<tag>.lock` (PID + 时间戳)

- PID 活着 → 锁有效（无超时，修复可能跑数小时如 `zypper dup`）
- PID 死了 → 残余锁，自动清（处理 SIGKILL / 崩溃残留）
- 检测: `kill -0 <PID>`（不发信号，只查存活）
