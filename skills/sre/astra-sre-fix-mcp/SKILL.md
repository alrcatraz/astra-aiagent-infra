## 触发条件

- `diagnose system` 显示内存不足（空闲 < 1GB）
- `diagnose services` 显示 MCP 进程数异常（> 10）
- 直接发现某 MCP 已经停用但仍占用内存
- MCP 健康检查持续失败（如 uvx 依赖被阻断）

## 诊断流程

### 第一步：审计 MCP 进程

```bash
# 查看所有 MCP 进程
ps aux | grep mcp-server | grep -v grep

# 统计数量
pgrep -af mcp-server | wc -l

# 查看内存占用
ps aux --sort=-%mem | grep mcp-server | head -10
```

### 第二步：检查 MCP 注册状态

```bash
# 查看 Hermes 已注册的 MCP
hermes mcp list

# 对比运行中的进程和已注册的 MCP
# 不在列表中的进程是残余进程
```

### 第三步：检查是否有冗余

```bash
# 检查是否有功能重复的 MCP
# 例：SearXNG 既有 MCP 又有 web_search 直接配置
hermes mcp list | grep -i searxng
grep search_backend ~/.hermes/config.yaml
```

## 修复步骤

### [L2] 清理残余 MCP 进程

```bash
# 1. 杀死所有指定 MCP 进程
pkill -f "mcp-server-searxng" || true

# 2. 从 Hermes 移除 MCP 注册
hermes mcp remove searxng

# 3. 从服务注册表删除
psql -h 127.0.0.1 -U astramcp -d astra_kb -c "
  DELETE FROM mgmt.services WHERE name LIKE '%searxng%';
"
```

### [L2] 修复健康检查假阳性

修改 MCP 的健康检查方式，移除对外部网络（如 pypi）的依赖：

```bash
# 坏：依赖 pypi
uvx --with mcp-server-time -- mcp-server-time --help

# 好：本地进程探活
pgrep -f 'mcp-server-time' > /dev/null
```

## 验证

- `pgrep -af mcp-server | wc -l` → 数量合理
- `diagnose system` → 空闲内存恢复
- `hermes mcp list` → 只显示有效的 MCP

## 已知陷阱

| 陷阱 | 说明 |
|:----|:------|
| MCP 进程会累积 | Hermes Agent 集成模式下每次请求可能启动新进程，需定期审计 |
| 功能可能重叠 | 部署新 MCP 前先检查 web_search 等工具是否已有相同功能 |
| 删除前先备份配置 | 先备份 `config.yaml` 再操作 `hermes mcp remove` |
