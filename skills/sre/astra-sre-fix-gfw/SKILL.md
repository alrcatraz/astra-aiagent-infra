## 触发条件

- `diagnose network` 显示多个外部服务同时 HTTP 000（超时）
- DNS 解析返回 `198.18.0.0/15` 或 `198.19.x.x` 范围内的 IP（GFW DNS 投毒签名）
- pypi.org、api.github.com、1.1.1.1 等国际服务不可达，但国内服务正常
- 健康检查报告多个对外 API 同时宕机（实际可能是网络层问题，非服务故障）
- **所有海外 HTTPS 站点都慢**（TLS 握手普遍 &gt;1s），但国内站点正常 — 提示代理隧道开销
- **Matrix 消息延迟明显增加**（从发消息到收到已读回应 &gt;2s），且不限于加密/解密问题
- **Gateway sync 错误异常增多**（&gt;20次/天），错误信息为空字符串（`asyncio.TimeoutError`）

## 诊断流程

### 第一步：确认 GFW 特征

```bash
# 检查 DNS 投毒：解析到 198.18.x.x 就是被投毒了

# 检查多个独立服务是否同时失败（mass failure pattern）

curl -s --connect-timeout 5 -o /dev/null -w '%{http_code}' https://api.github.com
curl -s --connect-timeout 5 -o /dev/null -w '%{http_code}' https://1.1.1.1
```

### 第二步：检查是否是周期性窗口

```bash
# 查看健康检查历史中的失败时间线
psql -h 127.0.0.1 -U astramcp -d astra_kb -c "
  SELECT checked_at::text, service_id, status

```

GFW 干扰常有约 1 小时周期。如果多个独立服务在完全相同的时间窗口同时失败，是网络层问题而非服务故障。

## 关联诊断模式：Matrix Gateway Sync 超时

Gateway 的 Matrix 同步长轮询（`_sync_loop`）有 45s `asyncio.wait_for` 兜底。
当连接不稳定时，会出现持续的 `sync error:  — retrying in 5s` 日志（空错误信息
= `asyncio.TimeoutError`）。

### 诊断步骤

```bash
# 1. 按天统计 sync 错误数
grep "sync error" ~/.hermes/logs/gateway.log | sed 's/ .*//' | sort | uniq -c

# 2. 正常基线：~9次/天（仅正常长轮询超时）
# 异常信号：>100次/天，且每 ~72s 循环出现

# 3. 排除服务端原因
ssh vps-hk '
  # 检查 Nginx proxy_read_timeout
  grep proxy_read_timeout /etc/nginx/snippets/synapse-proxy.conf
  # 正常值: 3600 (1小时，足够了)
  
  # 检查 Synapse worker 状态
  systemctl is-active matrix-synapse-worker1
  
  # 检查 Synapse 日志中是否有 sync 相关错误
  tail -100 /var/log/matrix-synapse/homeserver.log | grep -iE "sync|timeout|warn|error"
'
```

### 根因分析

当服务端配置正常（Nginx timeout=3600s、Synapse worker active）、但
sync 错误暴增时，根因在 **网络路径可靠性**：

```
常见场景：
  Gateway → Cloudflare(7s 延迟, 50% 卡死) → VPS-HK → Synapse
  
  替代方案：
  Gateway → EasyTier(0.14s, 零卡死) → VPS-HK → Synapse
```

修复方式：让 Gateway 通过更稳定的网络路径连接 Synapse。可以用
`/etc/hosts` 将 homeserver 域名指向内网 VPN IP（如 EasyTier 10.20.4.10），
或使用多路优选脚本自动选择最优路径。

### 验证修复

```bash
# 配置后再次统计 sync 错误
grep "sync error" ~/.hermes/logs/gateway.log | sed 's/ .*//' | sort | uniq -c
# 应降回 ~9次/天
```

## 关联诊断模式：PassWall 全局代理导致的 HTTPS 隧道开销

当 Router 上运行 PassWall（或类似透明代理）且设为 **全局代理模式**（`tcp_proxy_mode='proxy'`）时，所有海外 HTTPS 流量都会经过代理隧道（VLESS/Trojan/Hysteria2 等），增加一层加密转发开销。

### 诊断步骤

```bash
# 1. 使用 curl 的计时变量拆分 TLS 延迟
curl -s -o /dev/null -w \
  "TCP: %{time_connect}s  TLS: %{time_appconnect}s  总计: %{time_total}s\n" \
  --connect-timeout 10 https://matrix.gloriosa.space/_matrix/client/versions

# 2. 对比其他海外 HTTPS 站点（如果都慢 → 代理隧道问题）
for site in "github.com" "google.com" "cloudflare.com"; do
  tls=$(curl -s -o /dev/null -w "%{time_appconnect}" --connect-timeout 5 "https://$site" 2>/dev/null)
  echo "  $site: TLS ${tls}s"
done

# 3. 对照测试国内站点（正常 → 确认是海外流量问题）
curl -s -o /dev/null -w "baidu.com TLS: %{time_appconnect}s\n" \
  --connect-timeout 5 https://baidu.com

# 4. 检查 PassWall 是否为全局代理模式
ssh root@192.168.0.1 "
  echo '模式:' \$(uci get passwall.@global[0].tcp_proxy_mode)
  echo '启用:' \$(uci get passwall.@global[0].enabled)
  echo '节点:' \$(uci get passwall.@global[0].tcp_node)
"
```

### 诊断信号解读

| 模式 | curl TLS 耗时 | 结论 |
|:----|:-------------|:------|
| 海外直连（无代理） | ~0.05-0.3s | 正常 |
| 经代理隧道 | **~1.5-2.5s** | 代理隧道开销，所有海外 HTTPS 都受影响 |
| 国内站点 | ~0.02-0.1s | 不受代理影响 |

### 修复：添加 PassWall 直连分流规则

在 PassWall 的 `Direct` 分流规则中，把有问题的域名加入直连列表：

```bash
# OpenWrt 上执行
ssh root@192.168.0.1
uci add_list passwall.@shunt_rules[7].domain_list='matrix.gloriosa.space'
uci commit passwall
/etc/init.d/passwall restart
```

**效果：** 流量绕过代理隧道，直接走路由器 → 公网 → Cloudflare → 目标服务器。

### 进阶修复：通过 EasyTier 直连 Synapse（绕过 Cloudflare）

如果要进一步绕过 Cloudflare 的 TLS 开销，让 Gateway 的 Matrix sync 通过 EasyTier 连接 VPS 上的 Synapse：

1. 确认 VPS 上 Synapse 监听易梯接口（默认只绑 127.0.0.1:8008）
2. 在 /etc/matrix-synapse/homeserver.yaml 中添加 `bind_addresses: ['127.0.0.1', '10.20.4.10']`
3. 或在 Nginx proxy 中增加一个 server block 监听 EasyTier IP
4. Gateway 侧用 /etc/hosts 将 homeserver 域名指向 EasyTier IP

### 陷阱

| 陷阱 | 说明 |
|:----|:------|
| Ping 到 Cloudflare 可能被本地拦截 | 在透明代理模式下，ping 到 Cloudflare IP 可能返回本地响应（0.2ms 级别的 ping 是本地路由器回声，非真实网络延迟） |
| curⓘ 计时全域名对比是关键 | 单独测一个域名无法区分「目标服务器慢」和「代理隧道慢」。多个国际 HTTPS 站点同时有一致的 ~1.5-2.5s TLS 延迟 → 肯定是代理隧道 |
